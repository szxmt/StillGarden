## 当前重构 Goal：JS API / 状态层纯净期拆分

本阶段把“重构”本身作为唯一 feature。目标不是加新功能，而是在稳定 V0.2 基线下，把 API 请求层和状态管理层从 `script.js` 干净抽离出来。任何业务行为、接口路径、返回格式、UI 表现、数据写入路径都不能改变。

### Goal 名称

```text
Goal R1：拆出 Lunette Web API client 与 domain state 层
```

### 为什么现在做

- `script.js` 当前约 3900 行，虽然已经比最初变小，但仍承担 API 请求、状态管理、业务流程、DOM 渲染、事件绑定。
- 这个体量下继续加功能会让 blast radius 过大，改一个小功能也可能牵连 Chat / Moments / Archive / Timeline / More。
- 当前 V0.2 基线已经验证稳定，没有新功能混入；此时只做结构拆分，最容易判断问题来自哪里。
- 如果等到新功能开发时边做边拆，业务 bug 和重构 bug 会混在一起，排错成本更高。

### 绝对准则

- 不改变业务逻辑。
- 不改变 API 路径。
- 不改变 API 请求 body 和响应处理结果。
- 不改变 DOM 结构。
- 不改变 CSS。
- 不改变 UI 视觉和交互。
- 不改变 `sessions/prototype` 数据。
- 不改变 localStorage key 名称和数据格式。
- 不引入构建工具。
- 不引入框架。
- 不把 Web DOM 逻辑塞进 `shared/`。

### 允许修改

- `frontend/lunatte-v0.2/index.html`：只允许增加新 JS 文件加载顺序。
- `frontend/lunatte-v0.2/script.js`：只允许把 API 和状态逻辑替换成新模块调用。
- `frontend/lunatte-v0.2/shared/lunatte-core.js`：只允许移动或复用纯 helper，不改变现有对外行为。
- 新增 `frontend/lunatte-v0.2/shared/lunatte-api.js`。
- 新增 `frontend/lunatte-v0.2/shared/lunatte-state.js`。
- 必要时更新 `frontend/lunatte-v0.2/README.md` 和本文档。

### 禁止修改

- 不改 `server.py`、`server_config.py`、`server_storage.py`。
- 不改 `styles.css` 和 `css/`。

## 剩余拆分计划

当前不是“完全不用拆”，而是“第一轮底座拆分已收口，R1 先完成 API/state 纯净期拆分，之后再按功能边界拆”。目标是避免继续把新功能堆进 `script.js` / `server.py`，但也不为了形式拆出一堆空模块。

### 当前体量

- `script.js`：约 3900 行，仍是最大风险点。
- `shared/lunatte-core.js`：约 880 行，已承接 Web/App 可复用底层。
- `server.py`：约 2450 行，仍包含 provider、chat、archive、wake、moments、timeline、context、routes。
- `server_config.py`：约 180 行，边界清楚，暂不需要继续拆。
- `server_storage.py`：约 180 行，边界清楚，暂不需要继续拆。
- CSS 最大模块是 `20-chat-profile-dialogs.css`，约 1000 行；目前可维护，但后续如果继续改聊天信息页/资料页，应再拆。

### 拆分原则

- 新功能不能继续直接塞进大文件的任意位置；先找对应模块，没有模块就先建最小真实模块。
- 每次只拆一个功能边界，拆完必须能启动和回归。
- 不抽“只有一两个函数”的空壳模块，除非它解除一个明确依赖。
- 不为了 React Native / App 提前重写 Web 层；只把纯逻辑和数据协议往可复用层移动。
- 不把 DOM 节点、CSS class、toast、弹窗、事件绑定放进 `shared/`。
- `sessions/prototype` 仍是 V0.2 副本数据源，任何拆分都不能改变写入路径或 JSONL 追加语义。
