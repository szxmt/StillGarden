# Lunatte 全局总蓝图

更新时间：2026-07-01

这份文件是 V0.3 之后的全局状态板和总蓝图。它回答三个问题：

- 现在已经有什么。
- 哪些只是雏形或半成品。
- 之后按什么顺序继续往上建。

它不是替代所有专项文档，而是总入口。执行顺序看 `roadmap.md`，工程边界看 `engineering-rules.md`，记忆系统细节看 `memory-system-plan.md`。

## 项目一句话

Lunatte / 月亮小窝不是模型切换器，也不是普通聊天前端。它是一个本地优先、未来可扩展成真 App 的多住户关系系统。

核心目标：

- 林絮、噔噔、Aimas 是住户，不是按钮。
- 聊天、圈圈、Archive、Wake、时间线最终都回到同一套住户系统。
- 记忆是背景、证据、边界和时间线，不是剧本。
- 权限隔离要明确，但不能把住户拆成“聊天人格、评论人格、发圈人格”。

## 参考资料读后结论

### remax / Antigravity 魔改版

位置：

```text
D:\Aaa.项目\lunatte\remax-antigravity-archive
```

可吸收：

- “完整的人，不是功能模块”的系统原则。
- 所有入口最终进入同一条住户时间线。
- 先做壳，再做连续时间线，再做人格注入方式。
- Aimas 应走独立 Agent Connector，不混进普通 provider。
- 工具输出应该折叠为“小电脑工作卡片”，不要在聊天流刷终端日志。

不能直接用：

- 不直接覆盖当前 V0.3 代码。
- 不直接恢复旧目录结构。
- 不直接把 remax 的私有数据放入公开仓库。

### Tidal Echo

参考价值：

- PWA + VPS relay + 单密钥的私密通道设计。
- 手机端“添加到主屏幕”的低成本路线。
- HTTPS / Service Worker / 推送 / SSE 的实际部署约束。
- 前后端路径、密钥、缓存版本需要严格管理。

对 Lunatte 的修正：

- 可作为未来“手机入口 / relay / 推送”参考。
- 不能把它当最终方案，因为用户目标是真 App，不是只做 WebView 或 PWA。
- VPS 先做轻量桥，不承载完整私有记忆库。

### Memory Constellations

参考价值：

- 记忆分层：碎片 → 实体 → 叙事片段 → 故事弧线。
- 混合检索：FTS5 + 向量 + 实体聚合 + RRF。
- 自动抽取候选、再整理为长期叙事。
- 热力图和星图依赖稳定事件模型，不是先做 UI。

对 Lunatte 的修正：

- 不能照搬单伴侣系统，Lunatte 的核心是多住户隔离。
- 自动抽取必须先进入候选/审计，不直接写长期记忆。
- 先做 SQLite + FTS5 副本层，再考虑向量和叙事层。
- 星图/热力图不应压过手机端主体验。

## V0.3 之后的总路线

### Phase 1：总蓝图与当前状态收口

目标：

- 当前状态、总蓝图、执行路线清楚。
- 后续每轮只改对应文件，不再反复重写所有文档。

完成标志：

- `全局总蓝图.md` 存在。
- README 指向它。
- `roadmap.md` 只负责下一步执行顺序。

### Phase 2：UI 试点，不全量重写

目标：

- 先验证 token/components 能支撑正式 UI。
- 不为每个页面复制主题 CSS。

推荐试点顺序：

1. Chat 请求预览 UI 化。
2. Chat 信息页 / Profile。
3. Archive 记忆编辑/审计页。
4. Moments 卡片密度。
5. Timeline / 热力图。

完成标志：

- 新 UI 不改 API。
- 不改数据结构。
- 不新增大单体 CSS。
- 试点可复用到其他页面。

### Phase 3：统一事件模型

目标：

- 给聊天、圈圈、评论、Wake、确认记忆、Archive 操作建立统一事件口径。

先设计，不急迁移。

最小字段：

```text
event_id
source
actor
target
event_type
text
created_at
visibility
sensitive
related_room
raw_ref
```

完成标志：

- `/api/timeline` 的事件格式稳定。
- JSONL 到事件 schema 有明确映射。
- 不破坏现有数据。

### Phase 4：SQLite + FTS5 副本索引

目标：

- 保留 JSONL 原始真源。
- 建立 SQLite 副本索引，支撑搜索、聚合、热力图和 RAG。

顺序：

1. 建本地 SQLite 文件，不提交。
2. 导入 JSONL 副本。
3. 建 FTS5。
4. 对齐 Timeline 查询。
5. 对齐 Archive 搜索。
6. 输出索引健康状态。

完成标志：

- 删除 SQLite 后可从 JSONL 重建。
- 查询结果可解释。
- 不直接替代 JSONL。

### Phase 5：Archive 治理

目标：

- 让长期记忆可编辑、可撤回、可审计。

顺序：

1. confirmed-memory 编辑。
2. 撤回 / 软删除。
3. 权限变更记录。
4. 每日回声。
5. 周摘要。
6. 时光胶囊。

完成标志：

- 长期记忆不是只增不改。
- 每次改动有审计记录。
- RAG 能过滤撤回/敏感/不可读内容。

### Phase 6：住户整体人格与请求包治理

目标：

- 不是给每个功能写一套人格 prompt。
- 所有入口都回到同一个住户系统。

要做：

- 林絮、噔噔、Aimas 的请求包统一结构。
- 明确短期窗口、长期候选、确认记忆、self、shared、禁止读取列表。
- 前端可查看“这次实际送出了什么”。
- 减少过厚 system 包装，避免模板任务感。

完成标志：

- Chat / Moments 评论 / Wake 用同一套住户上下文构造原则。
- 住户不是旧记录模仿器。
- provider 只是通道，不是人格本体。

### Phase 7：Moments / Wake 自动性

目标：

- 自动性先进入候选和收件箱，不直接打扰用户。

顺序：

1. 自动评论后台触发器。
2. 自动发圈候选。
3. Wake 自动候选。
4. 冷却 / 安静时段。
5. 通知。
6. 全部写入时间线。

完成标志：

- 自动行为可回看、可撤回、可审计。
- 不会刷屏。
- 不会跨权限读取。

### Phase 8：真 App 与低成本部署验证

目标：

- 不是 WebView 套壳。
- 先验证技术路线，不急正式上线。

顺序：

1. React Native / Expo 可行性验证。
2. API 协议和设计 token 复用评估。
3. Android APK 侧载。
4. PWA / Tidal Echo 风格 relay 作为过渡参考。
5. VPS 轻量同步桥。
6. 远程访问鉴权。

暂缓：

- 商店发布。
- Supabase 正式迁移。
- Cloudflare/VPS 正式公开服务。
- 把完整私有记忆库放云端。
