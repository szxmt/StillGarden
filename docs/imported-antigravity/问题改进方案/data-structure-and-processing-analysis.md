# 数据结构与后端处理逻辑：效率与设计缺陷分析

> **目标**：响应 `/goal` 深度挖掘指令，针对 `gemini` 和 `gpt` 文件夹下的日志解析逻辑，以及 `.jsonl` / `.csv` 数据存储格式进行全面诊断。评估其是否能支撑未来 `Abyssal Letter` 级别的“高级检索”与“热力图（Heatmap）展示”。

## 1. 致命的检索效率缺陷 (以 `memory_search.py` 为例)

通过深度审查 `tools/memory_search.py`，我们发现了极度落后且存在性能瓶颈的搜索逻辑：
- **全表扫描 (Full Table Scan)**：每次执行记忆搜索，后端都会通过 `read_jsonl()` 把庞大的 `.jsonl` 文件一行行读取并全部塞进内存，然后再用 Python 代码去遍历比对！这在数据量达到数万条时，会导致接口响应时间飙升至数秒甚至卡死。
- **原始的停用词过滤**：代码里硬编码了 `stop = {"我", "你", "她", "他", "的"...}`，然后用正则分词。这种做法不仅愚蠢，而且根本无法理解语境。
- **结论**：**目前的检索逻辑绝对无法支撑高级搜索。** 在后续重构中，必须引入轻量级的嵌入式数据库（如集成全文本搜索插件 `FTS5` 的 `SQLite`）或者直接上轻量级向量数据库（如 `ChromaDB` / `Qdrant`）来实现毫秒级语义检索。

## 2. 数据格式与热力图的不兼容性 (Timestamp 解析地狱)

观察 `parsed/daily_companionship_index.jsonl` 中的数据结构：
```json
{"source": "gemini", "timestamp": "2026-06-13 19:33:30 JST", "title": "Gemini Apps", ...}
```
再看 `tools/build_gemini_activity_summaries.py` 里的疯狂正则：
```python
r"([0-9]{4})\u5e74([0-9]{1,2})\u6708([0-9]{1,2})\u65e5\s+([0-9]{1,2}):([0-9]{2}):([0-9]{2})\s+([A-Z]+)"
```

- **格式极度脆弱**：时间戳被硬生生地存储为包含时区、中文年月日的“展示用字符串”，而不是标准化的机读格式。
- **为什么它不支持热力图 (Memory Heatmap)？**：
  要渲染类似于 GitHub 绿点图的 Memory Heatmap，前端需要的数据是一维的日期分布矩阵（比如 `{ "2026-06-13": 5次 }`）。而目前的数据格式要求前端或后端必须遍历成千上万条记录，并对每一条记录执行复杂的字符串日期解析。这不仅慢，还极易出错。
- **重构方案**：在数据入库前，必须将所有时间转换为标准的 **UNIX 时间戳 (Unix Epoch Time)** 或绝对的 **ISO-8601 (UTC)** 格式。前端直接按天进行 `Math.floor(timestamp / 86400)` 取模即可瞬间绘制热力图。

## 3. 冗余的双份存储

目前每个处理结果同时存了一份 `.jsonl` 和一份 `.csv`（如 `activity_index.jsonl` 与 `activity_index.csv`）。这在原型验证期可以容忍，但在生产环境下，这不仅浪费一倍的磁盘 IO，还增加了数据一致性的风险。

## 总结

现有的数据持久层就是个“草台班子”。为了匹配 `Abyssal Letter` 那种丝滑、瞬间响应的高级体验，在重写后端路由（FastAPI）的同时，我们必须废弃现有的 `.jsonl/.csv` 纯文本堆砌模式，将记忆档案迁移至真正的数据库或向量化存储引擎中。
