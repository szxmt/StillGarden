# 小窝软件化路线图 v0.1

这个文件记录“从本地原型到真正软件”的路线。

当前判断：

```text
梦想可以很远，但路要一段一段铺。
```

不要一开始就做：

- EXE 安装包
- 手机 App
- VPS 后端
- 公网域名
- 多端同步
- 主动唤醒

这些都可以做，但不应该是第一步。

## 当前阶段：本地小窝

状态：

- 已有本地记忆库。
- 已有 Alice / 林絮 与噔噔的隔离规则。
- 已有 Aimas 未来住户门牌。
- 已有静态小窝前端。
- 已有本地服务桥接 `memory_context.py`。

当前目标：

- 先把本地小窝变成“每天真的能打开”的东西。
- 不急着联网。
- 不急着安装包。
- 不急着移动端。

下一步建议：

- Home 的数字接真实索引。
- “生成上下文草稿”继续美化。
- 做一个小手机聊天页原型。
- 增加本地启动说明和状态检查。

## 第一阶段：本地可用版

目标：

```text
在电脑上能稳定打开、查记忆、生成上下文草稿。
```

形式：

- 本地文件夹
- 本地网页
- 本地 Python 小服务
- 不联网
- 数据全部留在 D 盘

优点：

- 隐私最好。
- 不怕 VPS 配置问题。
- 不怕域名和证书。
- 适合继续整理记忆库。

完成标志：

- 打开小窝后能看到真实统计。
- 林絮/噔噔/Aimas 房间边界清楚。
- 上下文草稿能稳定生成。
- 不会混读私有记忆。

## 第二阶段：桌面软件

目标：

```text
像普通软件一样打开，而不是手动点一堆文件。
```

可能路线：

- 用 Tauri 做桌面壳。
- 或用 Electron 做桌面壳。
- 打包成 Windows EXE。

建议：

- 优先考虑 Tauri。
- 体积比 Electron 小。
- 更像“真正软件”，不是浏览器套壳感那么重。

注意：

- 即使是 EXE，里面也仍然可以是本地前端 + 本地服务。
- 先不要做安装包，先做便携版。

## 第三阶段：手机入口

目标：

```text
手机上能看小窝、收通知、打开房间。
```

可能路线：

- 先做手机适配网页。
- 再考虑 PWA。
- 最后再考虑真正 App。

建议：

- 先做 PWA。
- 可以像 App 一样添加到手机桌面。
- 比真正原生 App 简单很多。

注意：

- 手机访问本地电脑需要网络方案。
- 如果只在家里用，可以局域网。
- 如果外面也想用，就需要 VPS 或穿透。

## 第四阶段：VPS 后端

目标：

```text
让小窝从外面也能访问，并支持推送/主动唤醒。
```

VPS 可以做：

- 反向代理
- HTTPS
- 用户登录
- 远程访问
- Bark 推送
- 定时任务
- 主动唤醒
- 多端同步

不建议一开始把完整私密记忆库直接放 VPS。

更安全的路线：

- 本地电脑保存完整记忆库。
- VPS 只做入口和轻量任务。
- 必要时只同步摘要或经过筛选的片段。

## 第五阶段：域名和网页版

目标：

```text
用自己的域名打开小窝。
```

域名可以做：

- `home.example.com`
- `stillgarden.example.com`
- `moon.example.com`

需要准备：

- 域名 DNS
- VPS
- HTTPS 证书
- 反向代理
- 登录保护
- 备份策略

安全提醒：

- 记忆库不是普通网页内容。
- 上公网前必须先做登录和访问控制。
- 不要让搜索引擎索引。
- 不要把原始聊天记录直接暴露在网页目录里。

## Aimas / Hermes 接入

Aimas 当前状态：

- 未来住户。
- 跑在终端里的 Hermes 开源智能体。
- 当前只留门牌。
- 不读取 Alice / 林絮 或噔噔私有记忆。

未来接入方向：

- 前端把消息发给本地 Hermes。
- Hermes 返回回复。
- 小窝负责显示、边界和归档。
- 如果需要记忆，先建立 Aimas 自己的记忆区。

不要做：

- 不要直接把 Alice / 林絮 记忆给 Aimas。
- 不要直接把噔噔记忆给 Aimas。
- 不要默认让 Aimas 读 `self/`。

## 推荐顺序

1. 本地小窝 v0.2：真实统计 + 上下文草稿展示优化。
2. 本地小窝 v0.3：小手机聊天页原型。
3. 本地小窝 v0.4：本地聊天保存到 Archive。
4. 桌面便携版：一键启动，不做安装包。
5. PWA 手机入口：先像 App 一样放到手机桌面。
6. VPS/域名：只在登录和安全边界完成后再上。
