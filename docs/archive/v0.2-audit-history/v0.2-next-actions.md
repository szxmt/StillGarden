## V0.2 全局维护合同

本节是后续所有新功能和重构的写入规则。目标不是“文件越多越好”，而是保证每次改动都有唯一合理落点，不继续形成新的单体文件。

### 一句话结论

- 全局视觉不在页面 CSS 里改；先改 token，再改 components，最后才改页面模块。
- 页面 CSS 只负责该页面独有布局，不复制按钮、输入框、开关、卡片、弹层、toast、头像等通用样式。
- `script.js` 只做启动；页面 controller 只管 Web DOM；可复用逻辑必须进入 `shared/`。
- 后端 `server.py` 只做 HTTP/static 入口；routes 只解析请求；services 只做业务；providers 只对接模型；未来 repositories 只管数据 IO。
- config 不能继续混成一坨：public config、secret config、feature policy 必须分开维护。

### CSS 维护方案

当前 CSS 已进入 token / foundation / components / page modules 四层。后续全局样式按以下顺序处理：

1. `css/00-tokens.css`
   - 只放设计 token：颜色、字体、字号、圆角、阴影、间距、层级。
   - 做主题化时优先改这里，不允许每个页面重写一套主题。

2. 未来可选 `css/01-theme-base.css`
   - 只做语义变量映射，例如 surface、text、border、accent、danger。
   - 不写页面选择器。

3. `css/05-components.css`
   - 收纳通用组件：按钮、输入框、select、textarea、开关、卡片、弹层、toast、头像、状态条、列表项、分页。
   - 同一种组件样式在 3 个页面出现时，必须抽到这里。

4. 页面 CSS
   - `20-chat.css` 只写 Chat 独有布局。
   - `21-profile.css` 只写 Profile 独有布局。
   - `22-dialogs.css` 写通用弹层/确认层。
   - `23-chat-search.css` 写 Chat 搜索和上下文预览。
   - `30-archive.css` / `31-timeline.css` 分别写 Archive / Timeline。
   - `50-moments.css` / `51-wake.css` 分别写 Moments / Wake。
   - 页面 CSS 不允许新增硬编码色值；需要新颜色时先补 token。

CSS 阈值：

- 超过 700 行：必须审查是否混入多个功能。
- 超过 1000 行：必须先拆再做新功能。
- 页面模块出现通用组件定义：必须移到 components 层。

### JS 维护方案

当前 `script.js` 已经是启动入口，R5 后 controller 已全部低于 700 行。本轮之后按以下边界维护：

- `shared/lunatte-core.js`：纯 helper、格式化、常量，不访问 DOM。
- `shared/lunatte-api.js`：唯一 API client，页面层不得直接拼 `/api/*`。
- `shared/lunatte-state.js`：browser storage、local state、纯状态计算，不访问 DOM。
- `web/*-controller.js`：页面入口、事件绑定、Web DOM 协调。
- `web/*-render.js`：页面渲染函数。
- `web/*-actions.js`：页面动作流程。
- `web/*-helpers.js`：仅该页面使用的轻量 helper。

JS 阈值：

- controller 超过 700 行：新增功能前必须拆。
- render 函数超过 120 行：必须拆 view helper。
- 同一交互模式在 3 个页面重复：抽成 Web component helper。
- `shared/` 内出现 `document`、`querySelector`、`classList`、toast、modal：视为边界错误。

### 后端与配置维护方案

当前后端已经拆出 routes/services/providers/repositories，后续数据库和 RAG 必须继续沿用这个边界：

- `server.py`：只保留静态服务、API 入口、启动逻辑。
- `routes/`：只做 path 分发、query/body 解析、status/payload 返回。
- `services/`：只写业务规则和用例编排。
- `providers/`：只写外部模型、Hermes/Aimas、Gemini、OpenAI-compatible 请求。
- `repositories/`：统一 JSONL、assets、config/secrets、未来 SQLite/FTS5 的数据 IO。
- `server_config.py`：只放路径、默认值、轻量 normalize。
- `config_service.py`：只编排 public config；secret config 在 `config_secrets.py`；feature policy 在 `config_policy.py`。

后端阈值：

- service 超过 500 行：新增功能前拆 policy/helper/repository。
- route 中出现 JSONL/SQLite/provider 调用：必须退回 service/provider。
- service 中同时出现 prompt 构造、provider 调用、数据写入、列表读取四类职责：必须拆。

## Goal R1：JS API / 状态层纯净期拆分

本阶段把“重构”本身作为唯一 feature。目标不是加新功能，而是在稳定 V0.2 基线下，把 API 请求层和状态管理层从 `script.js` 干净抽离出来。任何业务行为、接口路径、返回格式、UI 表现、数据写入路径都不能改变。

### Goal 名称

```text
Goal R1：拆出 Lunette Web API client 与 domain state 层
```

### 为什么现在做

- `script.js` 当前约 3900 行，虽然已经比最初变小，但仍承担 API 请求、状态管理、业务流程、DOM 渲染、事件绑定。
- 这个体量下继续加功能会让 blast radius 过大，改一个小功能也可能牵连 Chat / Moments / Archive / Timeline / More。
- 当前 V0.2 基线已经验证稳定，没有新功能混入；此时只做结构拆分，最容易判断问题来自哪里。
- 如果等到新功能开发时边做边拆，业务 bug 和重构 bug 会混在一起，排错成本更高。

### 绝对准则

- 不改变业务逻辑。
- 不改变 API 路径。
- 不改变 API 请求 body 和响应处理结果。
- 不改变 DOM 结构。
- 不改变 CSS。
- 不改变 UI 视觉和交互。
- 不改变 `data/sessions/prototype` 数据。

- Prototype JSONL：当前运行基准，继续保护，不能随手迁移。

- SQLite / FTS5：后续事件索引，不直接替代原始 JSONL，先做副本索引和审计模型。

数据库原则：

- 先 append-only event model，再做 edit/revoke/delete audit。
- SQLite 先做查询索引，不要一开始就成为唯一真源。
- 任何迁移都必须用 `data/sessions/prototype` 副本验证，不能直接写 V0.1 原始数据。
