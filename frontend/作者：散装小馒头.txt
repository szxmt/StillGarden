# 完整项目重构执行路线（Markdown 规整版）
## 整体执行铁则
严格按顺序执行，**前一步全量回归校验通过，才能开启下一步**；禁止穿插无关改造、新功能开发；所有拆分仅分文件、不改动业务逻辑、接口、视觉表现，拆分后必须全端回归测试。

## 第1步：CSS 模块化拆分
### 当前现状
单文件 `styles.css` 超大整合样式
### 目标目录结构
```
css/
├─ main.css        # 入口统一导入所有样式模块
├─ variables.css   # 全局主题Token、圆角/间距/色值变量（当前00-tokens.css移入此处）
├─ reset.css       # 全局样式重置、基础默认样式
├─ layout.css      # 页面通用布局、容器、弹性、栅格
├─ components.css  # 通用组件：按钮、弹窗、输入框、导航等
├─ pages.css       # 分页面专属样式：Chat/Moments/Archive/Timeline/Profile
└─ responsive.css  # 移动端、桌面端响应式适配规则
```
### 约束要求
1. 仅做代码分割与导入，**不修改样式逻辑、数值、选择器**；
2. 页面视觉无明显变动；首页正常渲染；
3. Chat / Moments / Archive / Timeline / Profile 五大页面无布局崩坏、错位。

## 第2步：CSS 视觉全量回归校验
拆分完成后立刻全平台核验，出现样式问题当场修复，校验项：
- 桌面端全页面布局、组件正常
- 移动端适配无错位、溢出、压缩变形
- 按钮、弹窗、输入框交互样式无错乱
- 顶部/侧边导航样式稳定
- Moments、Archive、Timeline 列表无排版偏移
全部修复完毕，才算CSS拆分阶段闭环。

## 第3步：JS 分层拆分
### 当前现状
单文件 `script.js` 整合全部逻辑
### 目标目录结构
```
js/
├─ app.js          # 全局入口、初始化、统一导出
├─ api.js          # 底层接口请求封装（最先拆分，最底层依赖）
├─ state.js        # 全局状态、本地存储、会话数据管理
├─ navigation.js   # 页面路由、跳转、页面切换逻辑
├─ chat.js
├─ moments.js
├─ archive.js
├─ timeline.js
├─ profile.js
└─ settings.js
```
### 拆分固定顺序（不可打乱）
1. `api.js`（底层接口）
2. `state.js`（全局状态）
3. `navigation.js`（路由导航）
4. 各页面业务脚本：chat / moments / archive / timeline / profile
### 约束要求
只做逻辑分文件，不新增、删除、修改功能代码；拆分后原有交互100%可用：
- 按钮点击、页面切换正常
- 本地数据读取、存储正常
- 后端API请求、响应流程不变

## 第4步：JS 功能回归测试
逐条核验核心业务功能，异常问题立即修复：
1. Chat 消息发送、渲染正常
2. Moments 列表加载、展示无异常
3. Archive 记忆库页面可正常打开
4. Timeline 时间线数据加载完整
5. Profile / 设置页面切换流畅
6. 配置项读写正常、错误提示不失效
全部校验通过，JS重构阶段完成。

## 第5步：后端 server.py 拆分（必须前端拆分全部完成后再执行）
### 当前现状
单文件 `server.py` 承载全部后端逻辑
### 目标目录结构
```
server/
├─ app.py        # 服务启动入口
├─ config.py     # 全局配置、路径、常量定义
├─ routes/       # 分模块接口路由
├─ services/     # 业务处理逻辑层
└─ storage/      # 文件、会话持久化读写层
```
### 拆分硬性约束
1. 所有API请求地址、返回数据格式完全不变，前端无需修改代码；
2. 仅做代码分层分文件，业务逻辑无改动；
3. 数据存储路径锁定 `V0.2/sessions/prototype`，禁止误写旧目录 V0.1；
4. 不会自动生成仓库根目录以外的文件。

## 第6步：后端服务回归校验
1. 服务可正常启动，无启动报错
2. 全部核心接口 `/api/stats` `/api/config` `/api/moments` `/api/timeline` 返回200正常响应
3. 数据读写路径锁定新版目录，无跨目录文件生成
全部校验通过，后端拆分闭环。

## 第7步：Archive 记忆库数据规范设计（核心数据层规则）
聚焦记忆编辑、撤回、删除、操作审计的数据可信规则，确定完整方案：
1. 记忆条目是否支持编辑
2. 编辑后原始历史记录如何留存（历史快照）
3. 编辑操作支持撤回机制实现方案
4. 删除逻辑：物理永久删除 / 软删除标记
5. 全量操作审计日志：记录操作人、操作时间、修改前后内容
方案落地后，Archive模块才具备正式记忆库使用能力。

## 第8步：SQLite + FTS5 检索索引层设计
### 当前存储现状
业务数据分散 JSON / JSONL / sessions / assets 文件夹
### 新增索引层方案（不迁移原始业务文件，仅做副本索引）
搭建SQLite数据库，启用FTS5全文检索，建立数据表：
- events 操作事件
- messages 对话消息
- moments 动态条目
- memories 记忆档案
- assets 附件资源
- audit 操作审计记录
### 目标能力
完成索引层后，实现：高速全文搜索、记忆精准检索、Timeline时间范围筛选、Archive多条件查询。

## 第9步：选择性吸收 AGY / Gemini 现有成熟方案
前提：项目整体分层结构全部梳理完毕再执行，禁止直接覆盖源码
### 可复用内容
全局CSS主题Token、组件基础样式、Profile页面优化逻辑、自定义下拉组件交互、通用UI交互思路
### 严格禁止操作
- 完整覆盖 `index.html` / `script.js` / `styles.css`
- 批量执行外部补丁脚本
- 直接复制整套UI模板作为项目源码
### 执行规范
单模块拆解：参考一块 → 适配本项目结构改写 → 本地测试验证 → 单独提交，分步落地。

## 第10步：参考外部UI模板重构自有视觉
UI模板仅作为视觉参考图纸，禁止直接复制源码：
1. 确定Lunette项目统一视觉风格定位
2. 筛选可复用布局、动效、组件设计思路
3. 基于项目现有数据结构、CSS变量体系**完全重写**，适配自有业务逻辑
全部重构完成后，再统一迭代整体UI视觉。

## 第11步：全项目功能清单梳理分级
梳理所有需求，分为6类清单：
- 已完成可稳定使用
- 半完成待补全
- 存在Bug需要修复
- 短期准备开发
- 中长期规划功能
- 永久搁置不开发
将所有新增想法统一录入清单，排定开发优先级，杜绝零散即兴开发。

## 第12步：正式开启新功能迭代开发
前置全部重构、规范、分层工作闭环后，再开发新功能，可落地内容：
- 高性能记忆检索
- Moments 自动推荐候选
- Archive 完整编辑/撤回/删除体系
- Profile 记忆批量管理
- Timeline 多维度筛选
- AI主动唤醒能力
- 精细化UI视觉迭代
- 全端移动端深度适配
- 一键部署流程优化
